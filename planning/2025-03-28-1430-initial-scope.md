# Initial Scope - Google Calendar Analyzer

Date: 2025-03-28 14:30
Status: Draft

## Objective

Build a minimal web app where users can:
- Sign in with Google
- Grant read-only Google Calendar access
- Query events by a user-specified time range
- View results in a table
- Export the result set to CSV for optional import into Google Sheets

## Tech Stack

- Backend: Go
- Templating: templ
- UI styling: Tailwind CSS
- Interactivity: htmx
- Hosting: single VPS deployment

## Product Scope (MVP)

### Included

- Google OAuth login flow
- Read-only Calendar API access (`calendar.readonly`)
- Date/time range input
- Events table render in web UI
- CSV export of current filtered results
- Multi-user support

### Deferred (Post-MVP)

- Keyword filtering (`contains`, `exact match`)
- Saved queries and user history
- Background sync/caching
- Advanced calendar selection and cross-calendar analytics

## Architecture Decisions

### 1) No database for MVP

Decision:
- Do not introduce a DB initially.

Rationale:
- Keep setup and operations minimal.
- Faster delivery for first usable version.

Tradeoff:
- No durable server-side user/session/query storage.

### 2) Stateless multi-user sessions via cookie

Decision:
- Use secure cookie-based session state per user.

Session payload (minimal):
- Access token
- Access token expiry time
- Minimal user identity fields (e.g., subject/email)

Security requirements:
- Cookie must be signed and encrypted.
- Cookie flags: `Secure`, `HttpOnly`, `SameSite=Lax`.
- Short cookie/session lifetime.

### 3) No refresh token in MVP

Decision:
- Do not persist refresh tokens at this stage.

Rationale:
- Reduces storage/security complexity.
- Fits "minimal and stateless first" constraint.

Expected behavior:
- When access token expires, user re-authenticates.
- This is acceptable for current phase.

### 4) Google Calendar data retrieval on demand

Decision:
- Fetch events live from Google Calendar API per query request.

Rationale:
- Avoid local event persistence/caching complexity.

Tradeoff:
- Repeated API calls for repeated queries.

## High-Level Request Flow

1. User visits app and clicks "Sign in with Google".
2. OAuth callback completes and session cookie is set.
3. User submits `start` and `end` date/time range.
4. Server calls Google Calendar API `events.list` with range filters.
5. App returns an HTML table (htmx partial update).
6. User clicks "Export CSV" and downloads matching results.

## Proposed Route Surface (MVP)

- `GET /` - landing page + query form + current session state
- `GET /auth/google/login` - initiate OAuth flow
- `GET /auth/google/callback` - exchange code and set session cookie
- `POST /events/query` - return events table partial (htmx target)
- `GET /events/export.csv` - stream CSV for same filters
- `POST /auth/logout` - clear session cookie

## Query Contract (Forward-Compatible)

Use one normalized query object in handlers:
- `start` (required)
- `end` (required)
- `q` (optional, future)
- `match_mode` (optional, future values: `contains`, `exact`)

MVP implementation can ignore `q` and `match_mode` initially while preserving compatibility.

## Security and Reliability Notes

- Validate OAuth `state` to prevent CSRF.
- Enforce UTC internally for filtering, render with explicit timezone display.
- Validate `start < end` and reasonable max date range.
- Handle Google API pagination to avoid truncated result sets.
- Handle token expiry gracefully: redirect to re-login flow.

## Deployment Model (VPS)

- Single Go binary
- Reverse proxy (Caddy or Nginx) for TLS termination
- Environment variables for secrets/config:
  - `GOOGLE_CLIENT_ID`
  - `GOOGLE_CLIENT_SECRET`
  - `GOOGLE_REDIRECT_URL`
  - `SESSION_SECRET`
  - `APP_BASE_URL`

## Exit Criteria for MVP

- User can sign in and sign out successfully.
- User can query events for a chosen period and see a table.
- CSV export matches table dataset for the same filter inputs.
- Expired session/token path is handled cleanly with re-auth prompt.
- App deploys and runs on a VPS with HTTPS.
